## Design DNA

Nokta Distillery should look like a precision instrument, not a note app and not a chatbot. The Stitch direction is a near-dark, highly structured, engineering-first UI with dense card surfaces, sharp hierarchy, compact operational controls, and restrained accent color. The system name and visual thesis generated by Stitch are:

- Design system name: `Obsidian Logic`
- Creative north star: `The Precision Monolith`

Core interpretation:

- calm, serious, technical
- mobile-first
- compact but not crowded
- card-based and process-oriented
- strong hierarchy over decorative visuals
- reviewable system UI, not conversational UI
- built to scan cleanly in a 60-second demo

Color usage rules:

- Use layered dark surfaces instead of flat black.
- Prefer surface contrast over hard borders.
- Avoid full-strength divider lines.
- Use a low-opacity ghost border with `outline_variant` only when crisp separation is needed.
- Use `tertiary` for active system states, running states, and key technical highlights.
- Use `error` and `error_dim` for contradictions, destructive actions, and conflict emphasis.
- Use `primary` gradient surfaces for the main CTA and the most important "actuator" actions.

## Typography

Typography extracted from Stitch:

- Display / headline font: `Space Grotesk`
- Body / label font: `Inter`

Typography rules:

- Screen titles and important section headers use `Space Grotesk`.
- Dense content, metadata, and readable operational text use `Inter`.
- Labels should feel instrument-like: uppercase or near-uppercase with slight tracking.
- Source IDs, run IDs, wiki paths, and numeric process data should use mono-style treatment or at least tabular alignment.
- Typography should communicate authority and legibility, not friendliness.

## Shape And Spacing

- Roundness token: `ROUND_FOUR`
- Spacing scale: `1`

Implementation interpretation:

- Medium-small corner radius. Do not use large soft radii.
- Comfortable page padding with tighter card internals.
- Card internals should be dense and efficient.
- Stitch guidance sets card internal padding around `0.75rem`.
- Use vertical spacing to separate groups instead of heavy dividers.
- Keep chips compact and keep metadata rows short.

## Component Rules

### Buttons

- Primary buttons should feel machined and prominent.
- Use a subtle `primary` to `primary_dim` gradient for the main action.
- Secondary buttons use ghost-border styling rather than heavy fill.
- Tertiary buttons are text-first and technical, not playful.
- Review actions should read like system operations, not social buttons.

### Cards

- Cards are the core visual container.
- Prefer tonal separation between background, card shell, and card header.
- Do not rely on horizontal rules for sectioning.
- Each card should feel like a "distillation chamber" or system module.

### Inputs

- Large text input areas use the darkest inset surface to feel etched into the system.
- Focus should sharpen the control with a more visible ghost border.
- Input should feel like a data buffer, not a casual notes field.

### Chips And Status

- Chips are compact operational metadata.
- Neutral chips can use `secondary_container`.
- Active or running states can use `tertiary`.
- Priority and conflict chips should be stronger and more urgent.

### Signature Feature Styling

Wiki Patch Preview is the signature feature and should be the strongest visual module on the Distilled Memory screen. It should look more like a proposed system patch than a generic list. Recommended treatment:

- brighter or more contrast-rich card shell
- more obvious path formatting
- action + path rows
- source trace and affected-card metadata
- clear pending review status

## Screen Layout Notes

### 1. Input

Layout intent:

- compact top header with strong product title
- subtitle directly under header
- primary large input card dominates the screen
- metadata chips sit near the input, not hidden
- action row clearly distinguishes primary process action from utility actions
- pipeline hint appears as a compact directional diagram

Required visible content:

- `Nokta Distillery`
- `Compile messy project fragments into trusted project memory`
- large messy-notes text area
- `Load Sample`
- `Clear`
- `Process with Nokta`
- chips for `Mode: Distill`, `Output: Memory Bundle`, `Review: Human Confirmed`
- pipeline hint: `fragments -> idea cards -> canonical memory -> wiki patch`

### 2. Processing

Layout intent:

- no generic loading spinner as the main visual
- top-level system status summary with run metadata
- ten-step visible pipeline in order
- one active step should be visually obvious
- progress should feel deterministic and instrumented

Required visible process steps:

1. Splitting source fragments
2. Grouping duplicates
3. Extracting durable claims
4. Building idea cards
5. Detecting contradictions
6. Creating canonical summary
7. Selecting role views
8. Building evidence ledger
9. Preparing wiki patch
10. Creating handoff pack

### 3. Distilled Memory

Layout intent:

- this is the densest screen and the main demo screen
- canonical summary anchors the top
- role tabs stay compact and technical
- multiple modules stack as reviewable system outputs
- Wiki Patch Preview gets the strongest emphasis

Required visible sections:

- Canonical Project Summary
- Idea Cards
- Role Tabs
- Contradiction Queue
- Evidence Ledger
- Ambiguity Budget
- Wiki Patch Preview
- Handoff Pack

Idea card content:

- title
- type
- confidence
- sources
- review state

Review actions:

- Confirm
- Needs Review
- Discard
- Lock

Visual differentiation rules:

- contradiction cards should use stronger conflict signaling
- evidence ledger should feel more tabular / traceable
- wiki patch should feel like a proposed memory update artifact

### 4. Clarify

Layout intent:

- clarification queue, not a survey form
- show remaining-question progress
- each card should explain why the question matters
- actions are operational and quick

Each question card must show:

- question
- reason
- blocked area
- priority
- related sources

Actions:

- Answer
- Skip
- Mark Needs Review

## Interaction Notes

- Motion should be minimal and purposeful.
- Processing states can animate subtly but should never feel playful.
- The user should always see structured progression, not vague AI magic.
- Review states should be easy to change without making the interface feel like a chat thread.
- Role tabs should switch perspective without changing the core canonical memory foundation.
- Clarification answers should imply local updates to affected memory items rather than full uncontrolled regeneration.

## Implementation Guidance

- Treat this Stitch design as the UI source of truth for implementation.
- Preserve the near-dark engineering-tool feel in React Native + Expo.
- Do not flatten this into a generic notes app layout.
- Do not style this like a chatbot.
- When implementing, keep the Wiki Patch Preview as the signature visual moment.
